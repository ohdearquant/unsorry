## [1.21.0] - 2026-06-17

### Added

- Added an operator preflight doctor (`python3 -m tools.repo.doctor`, [ADR-065](docs/adrs/ADR-065-Operator-Preflight-Doctor.md) / [SPEC-065-A](docs/adrs/specs/SPEC-065-A-Operator-Preflight-Doctor.md)) тАФ a first slice of the ADR-051/ADR-056 operator surface. It runs cheap, side-effect-free health checks and exits non-zero on any failure. The flagship `pr-token` check verifies the active token can actually open pull requests by POSTing to the pulls endpoint with no `head`/`base` (rejected at validation as HTTP 422 тЖТ never creates a PR, but only after authorization, so an unauthorized token returns HTTP 403 first). The scheduled `queue-dispatcher` workflow now runs it as a fail-fast preflight, so a `REFRESH_TOKEN` lacking pull-request-write fails the job in one step with a fix-it message instead of silently dispatching zero PRs and dying on `Resource not accessible by personal access token (createPullRequest)`. The check registry is extensible for future operator checks (runner capacity, last dispatcher run).

### Changed

- The queue dispatcher (`agent.sh --dispatch-queue`) now opens **at most one prove PR per goal** ([ADR-064](docs/adrs/ADR-064-Goal-Level-Dispatch-Deduplication.md) / [SPEC-064-A](docs/adrs/specs/SPEC-064-A-Goal-Level-Dispatch-Deduplication.md)). The coordinated-submission model (ADR-058) let the prove-time claim race (knowingly deferred by ADR-017) accumulate duplicate `queued/prove/<goal>/*` branches тАФ over half of a 653-branch queue covered only 316 distinct goals, and 26 goals were already proved on `main` тАФ so draining it would have spent ~56% of namespace verifier capacity on proofs that can never merge (first-merge-wins) and only close as conflicts. The dispatcher now skips a queued branch whose goal is already proved on `main` (authoritative `library/index` marker, read from a freshly-fetched `origin/main`), already has an open prove PR, or was already handled earlier in the same pass тАФ non-destructively, so a goal's sibling branches remain as fallbacks until one merges. This restores at the dispatch boundary the duplicate-work bound that ADR-017's PR-level hygiene provided before queue-mode moved duplicates off PRs and onto dormant branches. Recovers verifier capacity for *useful* proofs with no new runners; the upstream claim race and bulk pruning of existing redundant branches are tracked as follow-ups.
- The required `gate-a` kernel replay now **shards across N parallel runners** ([ADR-063](docs/adrs/ADR-063-Sharded-Gate-A-Kernel-Replay.md) / [SPEC-063-A](docs/adrs/specs/SPEC-063-A-Sharded-Gate-A-Kernel-Replay.md) ┬з5, #1909), after the `gate-a-shard-pilot` validated it on real runners. The single serial `gate-a-replay` job became three: `gate-a-replay-plan` (partitions the replay target set into N disjoint shards), a `gate-a-replay` **matrix** (one leg per shard, each restoring gate-a-prepare's oleans and kernel-replaying its slice тАФ skipped entirely when no library module changed), and `gate-a-replay-cover` (fails closed unless the plan succeeded and every shard is green, i.e. every olean replayed exactly once). This cuts the ~1-hour full-replay long pole to ~1/N wall-clock so Gate A keeps pace with the submission rate. Soundness is unchanged тАФ the partition is unit-tested disjoint+covering, shards share only the git SHA (leanchecker inputs stay locally-derived, ADR-049), and the daily `gate-a-full-replay` backstop is retained. The shard count is the operator knob `vars.UNSORRY_REPLAY_SHARDS` (default 8); the required context name `gate-a` is unchanged (ADR-058).

### Fixed

- Fixed the queue dispatcher (`agent.sh --dispatch-queue`) stalling for ~9 minutes per pass and opening **zero** PRs on a large queue. The ADR-064 goal dedup checked open-PR membership with a per-branch `gh ... --search` call, but the GitHub search API allows only **30 requests/min** тАФ across a ~283-branch queue at `UNSORRY_DISPATCH_LIMIT` тЙе 10 it exhausted that bucket and stalled the whole pass on retry backoff (`9 min тЙИ 283 ├╖ 30`). The check is now resolved from a **single** `gh pr list --state open` call (`dispatch_open_pr_goals`, core quota 5000/h) made once per pass and tested by in-memory string membership тАФ O(1) API cost regardless of queue size. Dedup behaviour is unchanged (proved-on-main, open-PR, and same-pass goals are still skipped); a full real dispatch pass now completes in seconds instead of timing out.
- Fixed the Gate A GitHub `.lake/build` cache running on **every** runner profile, including Namespace, because three `if:` guards referenced a non-existent `needs.detect.outputs.volume` output instead of the per-job `prepare_volume`/`audit_volume`/`replay_volume` (#1921). `'' != 'true'` is always true, so on Namespace profiles тАФ where the persistent `.lake` volume (ADR-046) already keeps oleans warm тАФ a ~320 MB per-commit cache entry was written every job, churning and **evicting the warm entries** that prevent cold builds. The guards now match the `use-github-cache` lines, so the GitHub cache runs only on non-Namespace profiles as ADR-046 intended. Also added a `::warning::` to each Namespace cache-diagnostics step when the volume misses, so a cold-build-causing miss is visible to operators rather than silent. (The structural cold-start levers тАФ automating ADR-041 archiving to bound the active library, and a nightly warm-seed тАФ remain scoped follow-ups in #1921.)
- Fixed the scheduled `queue-dispatcher` workflow opening only **one** PR per run instead of up to `UNSORRY_DISPATCH_LIMIT`. The workflow set `UNSORRY_DISPATCH_LIMIT=10` (and exposed it as a `workflow_dispatch` input) but invoked `agent.sh --dispatch-queue --once`; `--once` caps a dispatch pass to a single PR (documented behavior), so the limit was dead and the queued-proof backlog drained at ~1 PR per scheduled tick. The workflow now runs one pass without `--once` and sets `UNSORRY_GOVERNOR_WAIT=0` (single pass then exit), so each scheduled run dispatches up to the configured limit, still bounded by the ADR-058 governor (open-PR and in-flight Gate A caps). agent.sh behaviour is unchanged.
- Fixed the README and `swarm/README.md` still pointing at `./swarm/agent.sh --prove --once` as the launch command, predating the `swarm/run.sh` one-command governed launcher (ADR-058) that CONTRIBUTING already recommends. The README "Running an agent" quickstart now leads with `./swarm/run.sh` (the recommended governed flow тАФ a resilient prover plus a single metered dispatcher), notes `--prove --once` for a single cycle, and `swarm/README.md` lists `run.sh` in its file table тАФ so a new contributor isn't sent down the pre-dispatcher path.
- Reconciled `swarm/run.sh`'s "run exactly ONE dispatcher" guidance with the scheduled `queue-dispatcher` workflow (which postdates run.sh): when that workflow is enabled it already _is_ the one dispatcher, so launching `run.sh` adds a second. ADR-064 goal-level dedup keeps this mostly harmless тАФ both dispatchers read the same open-PR set and skip a goal already proved or already PR'd тАФ but two passes can still race to open a duplicate PR within the window before one is visible to the other (first-merge-wins then closes the loser as a conflict, wasting a Gate A slot). `swarm/run.sh`, `CONTRIBUTING.md`, `swarm/README.md`, and `docs/recovery.md` now all say: on a repo with the scheduled dispatcher active, run a prover only (`./swarm/supervise.sh --prove`) rather than `run.sh`; use `run.sh` for a standalone/forked deployment with no scheduled dispatcher. Surfaced by an end-to-end run.sh test that observed its local dispatcher and the scheduled one opening PRs concurrently in the same window.

## [1.20.0] - 2026-06-17

Headline: **back open for business тАФ resilient recovery after the "ruv sledgehammer".** A contributor's ~337 direct `prove(...)` PRs ("the ruv sledgehammer", [#1904](https://github.com/agenticsnz/unsorry/issues/1904)) flooded Gate A and stranded ~200 valid proofs behind a saturated verifier queue. The backlog is now cleared and the swarm is hardened so it can't re-form: the stranded/failed direct PRs were closed, ~318 valid proofs **re-routed onto the metered queue**, the orphaned CI runs cancelled, and attribution fixed (`ruv` тЖТ `ruvnet`). Around that we shipped a full **resilience-and-recovery** stack across v1.17.1тАУv1.20.0 тАФ admission control now **offers the re-route on close** so a flood self-heals, a `reroute_stranded.py` tool recovers a stranded proof without re-proving, a scheduled **`queue-dispatcher`** drains the queue governor-metered, a **`close-superseded`** sweep retires originals as their re-routes land, and a [recovery playbook](docs/recovery.md) documents it all. This release lands the capacity piece: **sharded Gate A kernel replay** (ADR-063, #1909) cuts the ~1-hour verification long pole ~N├Ч across parallel runners so verification keeps pace with submissions тАФ every olean still kernel-replayed exactly once. The doors are open again, and the queue drains itself.

### Added

- Added sharded Gate A kernel replay ([ADR-063](docs/adrs/ADR-063-Sharded-Gate-A-Kernel-Replay.md) / [SPEC-063-A](docs/adrs/specs/SPEC-063-A-Sharded-Gate-A-Kernel-Replay.md)) to attack the verification-capacity bottleneck (#1909). The full kernel replay is a ~1-hour serial pass over the whole library because `leanchecker` holds mathlib resident per process (two won't fit one runner) тАФ but the work is already chunked and embarrassingly parallel *across* runners. New `plan` and `replay-shard` subcommands in `tools/gate_a/parallel_modules.py` partition the replay target set into N disjoint, covering shards (reusing the existing scope logic verbatim, so a shard never checks a different set than a full replay would) and replay one shard per parallel matrix runner, cutting the long pole to ~1/N wall-clock. Soundness is preserved тАФ every olean is still kernel-replayed exactly once (the partition is unit-tested disjoint+covering, the matrix cover job fails closed on any non-green shard, and the daily full-replay backstop is unchanged), and shards share only the git SHA so `leanchecker`'s inputs stay locally-derived (ADR-049). Ships with the tooling + tests and a **non-required** manual `gate-a-shard-pilot` workflow that validates the matrix on real runners; promoting it into the required `gate-a.yml` is a pilot-gated follow-up (ADR-058).
- Added `swarm/run.sh`: a one-command launcher for the governed swarm (ADR-058). It starts the two halves of the coordinated `--prove` flow together тАФ one dispatcher loop (`agent.sh --dispatch-queue`) plus one resilient prover loop (`supervise.sh --prove "$@"`) тАФ sharing the shell's `UNSORRY_*` env and torn down together on exit, so an operator runs `./swarm/run.sh` instead of managing two commands. Args pass through to the prover. Run exactly one dispatcher; for more provers, start additional `supervise.sh --prove`. Added to the agent-lint shellcheck/bash-n gate and documented in CONTRIBUTING.

### Changed

- Made the `aisp-advisory` Gate B check incremental тАФ it now validates only the `.aisp` coordination records **changed in the PR/push** (via `git diff` against the base, with `fetch-depth: 0`), instead of re-running `npx aisp-validator@0.3.0` over **every** record in `goals/translations/decompositions/library/index/proof-runs` on each run. The per-file `npx` loop across thousands of immutable `proof-runs/*.aisp` was making this advisory check the slowest in the pipeline (~8 min); a typical PR touches a handful of records, so it now finishes in seconds. Behaviour is otherwise unchanged: same job name (`aisp-advisory`), still `continue-on-error` (never reddens, ADR-003 / issue #318), and it falls back to validating all records when there's no usable base diff (first push / shallow history). The swarm-contract validation is unchanged.
- Gate A now routes prepare/archive, audit, and replay work to dedicated Namespace profiles (`namespace-profile-unsorry-prepare`, `namespace-profile-unsorry-audit`, and `namespace-profile-unsorry-replay`) so operators can scale each verifier lane independently.
- The Gate A `.lake/build` GitHub Actions cache (ADR-045) now only runs on **non-Namespace** profiles (`if: needs.detect.outputs.volume != 'true'`). On a Namespace runner the `.lake` cache volume (ADR-046) already persists `.lake/build` across jobs and runs, so the GitHub cache was redundant churn тАФ a ~320 MB per-commit entry (keyed on `github.sha`) that thrashed the 10 GB repo cache and created one near-duplicate entry per PR. Gating it keeps the cache list small and entries reused efficiently. Trade-off: on Namespace the #801 exact-key build-skip no longer fires, but the warm volume makes audit/replay's library build a cheap incremental, so throughput is unchanged.

## [1.19.0] - 2026-06-17

Headline: **the proof-recovery pipeline goes automatic** тАФ the queued-proof backlog now drains itself. A scheduled **`queue-dispatcher`** runs the governor-metered dispatcher every 15 minutes (it was previously only an operator-invoked mode), so re-routed and queued proofs flow into PRs that Gate A re-verifies and auto-merges; a **`close-superseded`** sweep retires each stranded direct PR once its goal lands, so the backlog shrinks instead of doubling; and **`docs/recovery.md`** (linked from a new README **Recovery** section) documents the whole submission/recovery machinery and the Gate A capacity backstop (ADR-058). Together with the earlier re-route tool and admission self-heal (v1.18.0 / v1.17.1), this closes the loop on "the ruv sledgehammer fix" (#1904) тАФ stranded proofs recover into the metered queue with no manual PR surgery and no lost work.

### Added

- Added `tools/repo/close_superseded.py` and a scheduled `close-superseded` workflow that closes a stranded direct `prove(...)` PR **once its goal is proved on `main`** (its queued re-route, or a peer, landed) тАФ and never a PR whose goal is still open. This is the safe, incremental implementation of "close the stranded originals once their re-routes land" for the ruv-sledgehammer cleanup (#1904) and the capacity drain (#1909): as the `queue-dispatcher` lands re-routed proofs, this sweep retires the superseded originals, so the backlog shrinks instead of doubling. Closing needs only the default `GITHUB_TOKEN`. Pure helpers are pytest-covered; `--author` and `--dry-run` are supported.
- Added `docs/recovery.md` and a **Recovery** section to the README documenting how the swarm submits proofs through the governor-metered queue, why a proof can get stranded (the direct-submission cutover + Gate A flood, #1904/#1909), and the self-healing recovery pipeline: the admission re-route offer, `reroute_stranded.py` (re-package without re-proving), the scheduled `queue-dispatcher`, and the `close-superseded` sweep тАФ plus the operator quick-reference and the Gate A capacity backstop (ADR-058).
- Added a scheduled `queue-dispatcher` workflow that drains the queued-proof backlog automatically. The `swarm/agent.sh --dispatch-queue` dispatcher (SPEC-007-A) previously ran only when an operator invoked it by hand, so `queued/prove/*` branches accumulated unbounded (the backlog in #1909). The workflow now runs it every 15 minutes (governor-metered via `UNSORRY_MAX_OPEN_PROVE_PRS` / `UNSORRY_MAX_GATE_A_IN_FLIGHT`, ADR-058), re-packaging already-locally-verified queued branches into PRs that Gate A re-verifies from scratch. It authenticates with the admin `REFRESH_TOKEN` (a GITHUB_TOKEN-opened PR would not trigger Gate A) and degrades to a report-only notice when that secret is unset. Self-throttling: it dispatches only while in-flight Gate A work is below the cap and no-ops when full.

### Added

- Added `tools/repo/reroute_stranded.py`, a maintainer tool that re-routes a stranded direct proof PR through the accepted queued path (ADR-058) without re-proving. Direct `prove(...)` PRs are no longer accepted (the queued-proof cutover), and pre-cutover ones strand with `gate-a-audit`/`gate-a-replay` perpetually queued. Self-contained proofs re-route cleanly; one importing a now-archived `Unsorry.*` module fails Gate A on dispatch (the safe outcome). Built to recover @ruvnet's stranded proofs (issue #1904, "the ruv sledgehammer fix").
- Added a **Sourcing** view to the leaderboard page ([ADR-060](docs/adrs/ADR-060-Contributor-Goal-Sourcing-Skill.md)): a fourth toggle that surfaces the sourcing leaderboard (`docs/metrics/sourcing-leaderboard.json`) in the browser тАФ who **sourced** each goal, ranked by goals sourced with a proved/open/difficulty split. ### Changed

## [1.17.0] - 2026-06-17

In CI every olean is rebuilt from the PR's sources, so a module whose source and entire import closure are unchanged rebuilds byte-identically to the olean already kernel-replayed when it merged; only changed modules and anything importing them (incl. their generated `*Binding` modules, which import their base) can differ. Result: a typical PR's replay drops from ~20 min / 12 GB-plus-swap to **~10тАУ30 s / <2 GB**. Conservative full-replay fallbacks cover push-to-`main` (the post-merge backstop), an uncomputable diff, and global-impact changes (`lean-toolchain`/`lakefile`/`lake-manifest`/`tools/gate_a/**`/the gate-a workflow). Unit-tested (`test_parallel_modules.py`); amends SPEC-006-B's full-library replay scope.

- Gate A's axiom audit now runs serially (`--jobs 1`). Each `axiom_audit` process holds a full mathlib image (~6тАУ7 GB), so two concurrent ones peaked ~13 GB on a 16 GB runner тАФ not an OOM, but enough to evict the `.olean` page cache, and the re-reads showed up as high CPU I/O wait (the parallel run thrashed rather than ran faster). The `--jobs` knob is retained for runners with materially more RAM (a 32 GB profile can raise audit back to `--jobs 2` and re-parallelize replay). SPEC-006-B updated.
