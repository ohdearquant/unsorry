## [1.31.0] - 2026-06-22

### Added

- An advisory `pr-changelog` workflow now reminds contributors when a user-facing harness change lands without a `changelog.d/` fragment (ADR-040, #445). It is advisory only — it emits a warning annotation and a job-summary note, never blocking the PR — and is swarm-safe by construction: it fires only on the harness surface (`tools/repo.pr_scope`) for user-facing conventional-commit types, so the swarm's proof PRs never trigger it.
- **Model → Pokémon registry & the first swarm operational task** (ADR-083). A new `swarm/housekeeping.sh` work package — run first by `run.sh` — researches each model in the leaderboard distribution (open/closed source, publisher, country, parameter size, canonical link) and assigns it a unique Pokémon identity, published to `docs/metrics/model-registry.json` for the guild frontend. One Pokémon per model per PR (labelled `model-registry`/`chore`), enforced by a deterministic validator (`tools/model_registry`, stdlib + pytest) and the `model-registry-gate` workflow (schema · Pokémon validity · uniqueness · one-new-entry diff, with a deliberate full-reset exception). The registry ships empty and the swarm populates it: `run.sh` blocks on housekeeping until **every** model has a Pokémon before any proving begins, draining the unnamed list serially (one PR settled onto main before the next). Each entry records the naming model (`provenance.assigned_with`, e.g. `claude / opus`) and the owning swarm contributor (`provenance.contributor`).
- Added a plain-language **"How mathematics actually gets made — and the part we're automating"** section to the README homepage, aimed at readers who aren't mathematicians. It grounds the project's purpose concretely: mathematics is written as a theorem first and then *proven*, and the proof is the slow, manual, error-prone part (illustrated with Fermat's Last Theorem and the computer-verified Kepler conjecture); proof assistants like Lean made *checking* mechanical and built mathlib, but writing the proof is still human labour — which is the bottleneck unsorry automates by handing proving to a swarm of autonomous workers (AI agents plus a deterministic sympy solver) that coordinate through the repository via AISP, with the kernel re-checking everything. Closes the gap where the docs assumed the reader already understood why machine-checked mathematics matters.
- Added a **stale-failed prove-PR janitor** (`tools/repo/stale_failed_prs.py` + `stale-failed-pr-janitor.yml`) that rebases swarm `prove/*` PRs whose Gate A failed *only because the branch fell stale*. When a prove-PR is not merged quickly, `main` races thousands of commits ahead and seals once-live `library/Unsorry/*.lean` modules into archive packages (ADR-041/ADR-048); Gate A then rebuilds those archived modules in their *old* state on the stale branch and trips a `--wfail` bar `main` has since fixed — the PR's own proof is sound, but it fails on library state that no longer exists (2026-06-22: 22 valid prove-PRs ~6,300 commits behind, all green after one `update-branch`). This is the inverse of the dropped-gate janitor — there the required gate is *absent*; here it is *present and failed*, so that janitor deliberately leaves these alone. It conservatively rebases only the stale-failure signature: a non-draft `prove/` PR, not `DIRTY`, with a *failed* `gate-a` and at least `--min-behind` (default 200) commits behind base. That threshold is both the staleness signal and the loop guard — `update-branch` resets the branch to ~0 behind, so a just-rebased PR cannot re-qualify until `main` drifts ≥200 ahead again (hours later, after Gate A has re-run and auto-merge has fired if green); a proof that still fails while *current* is treated as a real failure and never touched. Runs as `REFRESH_TOKEN` so the synchronize event re-dispatches Gate A (the default `GITHUB_TOKEN` would not), report-only when unset. Hourly backstop + manual dispatch; pure decision predicates unit-tested (16 new).

Makes sourcing safe for the decentralised "anyone runs `run.sh`" model; manual `sourcing.sh` is unchanged. New gate logic is unit-tested in the hermetic `--self-test`.

## [1.30.0] - 2026-06-22

### Added

- Added an idempotent attribution-relabel sweep (`tools/repo/relabel_attribution.py` + `attribution-relabel.yml`) — the maintained replacement for the one-shot PR #3218, which could not survive the live corpus (it conflicted and was always incomplete). ohdearquant's `mac-158f` deterministic sympy-template pipeline records proofs as `provider≜claude; model≜template-*`, overstating LLM involvement; the honest label is `provider≜python; model≜sympy`. The sweep rewrites every record carrying all three signals (`agent≜mac-158f` + `provider≜claude` + `model≜template-*`) and no-ops once corrected — self-healing as the pipeline produces more (~1,127 records to correct at first run). Precise and conservative: genuine LLM proofs by the same agent (e.g. `model≜sonnet`), `seedkit` fixtures, and other contributors are untouched; `solver≜` credit never changes (ranking unaffected). It edits provenance metadata only (soundness-neutral). Runs post-merge + hourly, committing to `main` via `REFRESH_TOKEN` with `[skip ci]` (report-only if unset). Pure `relabel_record` unit-tested.
- Added a **dropped-gate PR janitor** (`tools/repo/dropped_gate_prs.py` + `dropped-gate-janitor.yml`) that rescues PRs GitHub left stuck `BLOCKED` because it never dispatched their required `gate-a`/`gate-b` checks on the head SHA — a dropped `pull_request` event leaves the PR with auto-merge enabled but no required check able to report, and nothing re-triggers it (as happened to #3394 for ~7 h, 927 commits behind). It conservatively detects only that exact signature (a required gate with zero check-runs, none pending or failed, on a PR past a grace window and behind its base) and re-triggers the gates with `update-branch` — run as `REFRESH_TOKEN` so the synchronize event actually dispatches workflows (the default `GITHUB_TOKEN` would not), report-only when unset. Hourly backstop + manual dispatch; pure detection predicates unit-tested (15 new).
- Added the project origin story, *The Ten-Day Republic* (`docs/story.md`) — a narrative-nonfiction account of how unsorry went from a single commit on 10 June 2026 to a self-coordinating research institution (8,983 commits, 2,349 verified proofs, eight human contributors and an uncounted agent swarm) in eleven days, with figures taken from the repository's own history and quotes from Chris Barlow's public talk and contributors' messages. The README "story of unsorry" section gains a short summary linking to it.

### Changed

- The targets-board and proofs-visualisation post-merge refresh workflows now regenerate **once** and retry only the push (rebasing the disjoint docs commit onto the latest `main`), matching the leaderboard livelock fix. Their regeneration is sub-second so they were not affected in practice, but adopting the same shape keeps the three artifact refreshers consistent and prevents a future slowdown from reintroducing the multi-hour staleness seen on the leaderboard (follow-up to #426 / #3546).
- The post-merge **leaderboard** refresh now recomputes the corpus **once** per run instead of twice (ADR-082 / SPEC-082-A). The workflow previously ran a standalone `tools.leaderboard --check` drift probe and then a separate `--write` — two ~10-min full-corpus recomputes per refresh over the growing active+archive proof set (ADR-041). Under the swarm's merge firehose that doubled the window in which `main` advances mid-refresh, so the published `docs/metrics/leaderboard-ui.json` (and the live unsorry-guild proofs graph that reads it) lagged the latest merges by ~25–30 min, and it wasted CI runner minutes. A new single-pass `tools.leaderboard --write-if-stale` mode computes the seven generated artifacts exactly once, writes them only on drift, and signals drift via its exit code (`1` = was stale and rewritten, `0` = already in sync) — sharing one staleness/write definition with `--check`/`--write` (DRY). The cheap-push-retry/rebase loop (#426), the trigger model, and the artifacts produced are unchanged; the change is soundness-neutral (generated human-facing artifacts only). Halves the hot-path compute and the mid-refresh staleness window. New CLI mode is unit-tested for drift/no-op/idempotency/mutual-exclusivity.

### Fixed

- The queue dispatcher's fair-ordering helper (`fair_dispatch_order`) no longer prints a `BrokenPipeError` traceback when the dispatch loop stops reading early — normal whenever the dispatch limit is reached or the submission governor pauses. The traceback was harmless (the dispatch decision was already made) but read as a crash in `run.sh` logs.
- Fixed the leaderboard's `generated_at` (in `docs/metrics/leaderboard-ui.json`) so it reflects data freshness. It was pinned to the latest recorded proof-*run* time, so a provenance change that records no new run — e.g. an attribution relabel — left the board showing a stale timestamp even though its data was current. It is now the committer time of the latest commit touching the board's source data (`library/index`, `proof-runs`, `goals`, `packages`, `contributor-aliases.json`), so it bumps on every refresh that follows a real change (proof merges and relabels alike). This stays deterministic — the refresh's own docs-only `[skip ci]` commit doesn't touch those paths, so `--check` never sees timestamp-only drift and the refresh does not churn (ADR-036). Falls back to the latest run time when git is unavailable (e.g. a non-repo test fixture).
- Fixed the leaderboard "Providers and Models" bars clipping their count off-screen. The top row's bar is scaled to 100% of the longest model's proof count, and it was `flex-shrink-0` at `width: 100%`, so it consumed the whole row and pushed the `whitespace-nowrap` "N proofs · X%" label past the right edge (most visible on `python / sympy`, which leads with 1,100+ proofs). The bar now sits in a `flex-grow` track while the count is `flex-shrink-0`, so the count always keeps its space and the bar fills only the remainder — robust across viewport widths (unlike a fixed width cap, which the longer models label would still overflow on narrow screens).
- The leaderboard's **proofs over time** chart (and the rest of the community board) no longer lags the swarm by 1–2 h. After #426 the post-merge refresh ran on a `*/10` schedule, but GitHub throttles that cron to roughly one run an hour, so the board sat hours behind the merge firehose while the push-triggered `targets-board` and `proofs-visualisation` refreshes stayed current. The refresh now triggers on the merge that changes its inputs — the same proven model as those sibling workflows — with the cron retained only as a backstop; the regenerate-once / cheap-push-retry loop from the earlier #426 follow-up keeps it conflict-free under the merge load that originally motivated moving off per-push (#3720).
- The community leaderboard's *proofs over time* chart and contributor stats no longer freeze for hours during a proof-merge burst. The post-merge refresh now regenerates the leaderboard **once** and retries only the (cheap) push, instead of re-running the ~2-minute regeneration on every push attempt — which, under the swarm's merge cadence, lost every non-fast-forward race and left the board stale while the faster targets-board and proofs-visualisation refreshes kept up (follow-up to #426).
- Fixed the attribution-relabel sweep, which had never once run: the `attribution-relabel.yml` workflow invokes `python3 -m tools.repo.relabel_attribution --apply .` (positional root, matching `tools.gate_b`/`tools.leaderboard`), but the script only accepted `--root`, so every run died with `error: unrecognized arguments: .` (exit 2) before relabelling anything — born broken in #3394 (the unit test exercised the `--root` form, never the workflow's `--apply .`). The script now accepts the root positionally, with a regression test asserting the workflow's exact argv parses. Unblocks the sweep so mac-158f's deterministic-template proofs are finally relabelled `provider≜python; model≜sympy` instead of overstating `claude / template-*`.
- Extended the attribution-relabel sweep to correct chat-bit-01's `claude-web` `template-zmod-decide` proofs, which were recorded as `provider≜claude` but are a pure Lean kernel `decide` over a finite `ZMod n` (no LLM, no Python/sympy) — the board was overstating LLM involvement. They are now relabelled `provider≜lean; model≜decide`, the honest engine. The sweep is now a small two-rule table (mac-158f → `python / sympy` per ADR-079; claude-web → `lean / decide`), each rule scoped to its agent so an identical `template-zmod-decide` shape under any other contributor (or `seedkit` fixtures) stays untouched; `solver≜` credit is never changed. Idempotent and self-healing as before (~248 records corrected at first run).

### Fixed

- Fixed `gate-a-archive` exhausting runner disk on wide metadata PRs (e.g. #3218's 1831-file attribution correction). `validate-changed` rebuilt every changed archive package — each running `lake exe cache get` (a multi-GB mathlib fetch/clone) — so a PR touching index records across all 32 packages ran out of disk (`No space left on device`). For a change confined to `library/index/*.aisp` (attribution/provenance metadata), the proof modules, lakefile, and toolchain are byte-identical — the ADR-048 provenance guard already proves it, and the `lake build` only re-checks packaging that a metadata edit cannot break. - Fixed Gate A failing on fork PRs because of a cosmetic comment step. - Fixed the proof graph wrongly listing the GitHub user who *merged* a proof as its **solver**. For the ~300 (mostly archived) goals with no recorded `solver≜`, the visualiser had been falling back to the squash-merge author — i.e. the dispatcher who landed the PR, not the person who solved it. The Solver column now shows the recorded AISP solver only (“—” when there is none, per ADR-023's never-guess rule), and a new **Merged by** column reports who landed the PR, so the two are never conflated.

## [1.28.0] - 2026-06-20

### Added

- Added **sharded fork goal selection** ([ADR-076](docs/adrs/ADR-076-Sharded-Fork-Goal-Selection.md) / [SPEC-076-A](docs/adrs/specs/SPEC-076-A-Sharded-Fork-Goal-Selection.md)) — Phase-2 step 2c, the claimless, coordination-free mitigation for **cross-fork** duplicate proving. Fork-native proving (ADR-068) ranks goals deterministically, so independent forks on different machines all pick the same top goal; #3164's goal lock only coordinates co-located (same-host) provers. In fork mode the prover now reorders its ranked candidate list to **prefer goals in its own shard** — `shard = cksum(agent-id) mod K`, `K = UNSORRY_FORK_SHARDS` (default 8) — falling through to the rest in rank order when its slice is dry. `cksum` (CRC-32) is cross-machine stable, so two uncoordinated forks compute the same map and prefer disjoint slices with no round-trip. It is advisory only (never a correctness input — the kernel, first-merge-wins, and the #3164 lock remain the backstops), preserves the ADR-010 affinity/gap order within each shard, and leaves the canonical write-access selection path byte-for-byte unchanged. Sharding *reduces* (not eliminates) cross-fork collisions — the residue is what the still-evidence-gated Phase-2 lease (SPEC-053-A §8.4) would remove if the ADR-070 metric ever justifies it.
- Added a stale `queued/prove/*` branch janitor (`tools/repo/stale_branches.py` + `stale-branch-janitor.yml`). Once a queued proof branch's goal is proved+archived, the dispatcher correctly skips it (opening it would be a duplicate PR, the #3077 conflict), but the branch lingers forever — thousands accumulate (observed ~1k+ confirmable-stale of ~2.9k total) and every dispatcher pass lists and re-checks them all. The hourly janitor deletes branches whose goal `goals/<id>.aisp` positively marks `proved`/`archived`, in bounded batches; it is conservative (an open or missing goal record is left untouched) and uses the default `GITHUB_TOKEN` (deletes refs only, no PRs). Pure predicates (`goal_of_branch`, `done_goals`, `is_stale`) are unit-tested.

### Fixed

- Fixed admission control auto-closing legitimate `unblock(...)` and `decompose(...)` coordination PRs (issue #3128). `pr_admission.py` classified any `feature/goal-*` branch as a post-cutover "direct proof PR", but ADR-009 coordination PRs live on `feature/goal-*` branches too — so they were closed every time, leaving blocked parents permanently un-unblockable (every agent re-opened an identical `unblock(...)` PR each cycle, all closed in the same minute). Direct proof submissions are now identified by the reliable `prove(` title or the dedicated `prove/` branch, not the `feature/goal-` prefix; coordination PRs pass through.
- Hardened the auto-archiver so one inconsistent goal can no longer poison a whole block (and stall all archiving). A recompose/shared proof can place a goal's statement-binding theorem in a module owned by another goal; if that module is left active or was archived in an earlier block, the cut package fails Gate A's binding validation in isolation ("no library module declares '<name>'" — e.g. `nat-sq-lt-two-pow-s2`, which broke block 0033). `select_block` now runs a CI-faithful pre-check (`_binding_unsatisfiable_in_block`, mirroring `check_statement_binding`'s non-private `theorem/lemma` resolution) and **defers the whole decomposition component** of any goal whose binding theorem would not be declared inside the package, iterating until the block is self-consistent. The rest of the block still cuts, so archiving keeps running.
- The proof-credit guard now runs on the universal prover chokepoint (`swarm/agent.sh:resolve_solver` setup), not only in `swarm/run.sh`. The canonical write-access path is unchanged (fork mode is default-off and additive); `--prove-local` remains the purely-local, no-submission option. Note: GitHub requires a one-time maintainer approval of a new fork contributor's first Actions run, after which it is hands-off.
- `swarm/run.sh` now sources on demand **by default** ([ADR-069](docs/adrs/ADR-069-Launcher-Demand-Driven-Sourcing-Arm.md) / [SPEC-069-A](docs/adrs/specs/SPEC-069-A-Launcher-Demand-Driven-Sourcing-Arm.md)): the one-command launcher gains a third background arm — a `sourcer` loop running `./swarm/sourcing.sh --if-pool-empty` (ADR-067) — alongside the dispatcher and prover, torn down with them on exit. So a single `./swarm/run.sh` now proves, dispatches, **and** replenishes the backlog exactly when (and only when) the prove pool runs dry — no second command or cron needed. v1.23.0 shipped the `--if-pool-empty` mechanism but left the launcher unwired, so the automatic, default-on behaviour was absent; this closes ADR-067's deferred follow-up at the launcher level, leaving the prove arm / `agent.sh` / ADR-017 untouched. The arm only *adds* automatic empty-pool top-up — explicit `./swarm/sourcing.sh` (no flag) still sources on demand at any pool depth, so making auto-sourcing the default never suppresses a manual source. Default-on; set `UNSORRY_SOURCE_ON_EMPTY=0` to omit the arm (e.g. a deployment topped up by a scheduled sourcing job) and `UNSORRY_SOURCING_WAIT` (default 300s) to tune the re-poll interval. `run.sh` also gains its first hermetic `--self-test`, wired into `agent-lint.yml`.

### Changed

- Gate A's kernel replay (`leanchecker`) is now **incremental on PRs** (ADR-033/SPEC-033-A): it replays only the library modules changed in the PR plus their transitive **reverse-import closure**, instead of the whole library. Measurements drove this — `leanchecker`'s cost scales with the import union of the module set (1 module ≈ 10 s / 1.4 GB; a 30-module chunk ≈ 127 s / 12.5 GB; ~180 modules OOMs a 16 GB runner), while a typical proof PR changes 1–3 modules and re-replayed the unchanged rest for nothing. In CI every olean is rebuilt from the PR's sources, so a module whose source and entire import closure are unchanged rebuilds byte-identically to the olean already kernel-replayed when it merged; only changed modules and anything importing them (incl. their generated `*Binding` modules, which import their base) can differ. Result: a typical PR's replay drops from ~20 min / 12 GB-plus-swap to **~10–30 s / <2 GB**. Conservative full-replay fallbacks cover push-to-`main` (the post-merge backstop), an uncomputable diff, and global-impact changes (`lean-toolchain`/`lakefile`/`lake-manifest`/`tools/gate_a/**`/the gate-a workflow). Unit-tested (`test_parallel_modules.py`); amends SPEC-006-B's full-library replay scope.

- Gate A's axiom audit now runs serially (`--jobs 1`). Each `axiom_audit` process holds a full mathlib image (~6–7 GB), so two concurrent ones peaked ~13 GB on a 16 GB runner — not an OOM, but enough to evict the `.olean` page cache, and the re-reads showed up as high CPU I/O wait (the parallel run thrashed rather than ran faster). One image at a time keeps the cache hot. This matches the already-serial `leanchecker` replay (both heavy kernel passes hold a fixed mathlib image, so concurrency hurts on a 16 GB runner). The `--jobs` knob is retained for runners with materially more RAM (a 32 GB profile can raise audit back to `--jobs 2` and re-parallelize replay). SPEC-006-B updated.
