# Leaderboard Automation

## Single Refresh Command

Prefer one command for all generated leaderboard artifacts:

```bash
python3 -m tools.leaderboard --write .
```

The matching check should detect drift:

```bash
python3 -m tools.leaderboard --check .
```

If new artifacts are added, wire them into both modes.

## Policy A: Regenerate In Each Proof PR

Each proof/decomposition/failure PR that changes `library/index/` or `proof-runs/` also regenerates leaderboard artifacts.

Pros:

- main branch leaderboard is fresh after merge;
- no scheduled bot required;
- contributor sees the leaderboard update.

Cons:

- generated docs churn in proof PRs;
- possible merge conflicts during concurrent proof runs.

## Policy B: Drift Check Plus Generated-Artifact PR

Proof PRs commit telemetry source records only. CI or a scheduled job regenerates leaderboard outputs and opens a separate PR.

Pros:

- proof PRs stay narrow;
- fewer generated-file conflicts.

Cons:

- leaderboard can lag;
- needs scheduled workflow or maintainer action.

## Recommendation

Start with Policy A unless generated artifact conflicts become noisy. If CI workflow edits are CODEOWNERS-protected, first implement generator `--check` locally and wire CI in a follow-up reviewed PR.

## Validation Matrix

| Change | Checks |
|---|---|
| `tools/leaderboard/**` | `python3 -m pytest tools/leaderboard -q` |
| `library/index/*.aisp` | `python3 -m tools.gate_b validate .`; leaderboard check |
| `proof-runs/*.aisp` | `python3 -m tools.gate_b validate .`; leaderboard check |
| `docs/metrics/*.json` | generated by `--write`; checked by `--check` |
| `docs/leaderboard.html` | generated or manually reviewed; fetches UI JSON |
| README preview image | regenerate and inspect rendered output |
