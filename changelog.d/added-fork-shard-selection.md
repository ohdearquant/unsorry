Added **sharded fork goal selection** ([ADR-076](docs/adrs/ADR-076-Sharded-Fork-Goal-Selection.md) / [SPEC-076-A](docs/adrs/specs/SPEC-076-A-Sharded-Fork-Goal-Selection.md)) — Phase-2 step 2c, the claimless, coordination-free mitigation for **cross-fork** duplicate proving. Fork-native proving (ADR-068) ranks goals deterministically, so independent forks on different machines all pick the same top goal; #3164's goal lock only coordinates co-located (same-host) provers. In fork mode the prover now reorders its ranked candidate list to **prefer goals in its own shard** — `shard = cksum(agent-id) mod K`, `K = UNSORRY_FORK_SHARDS` (default 8) — falling through to the rest in rank order when its slice is dry. `cksum` (CRC-32) is cross-machine stable, so two uncoordinated forks compute the same map and prefer disjoint slices with no round-trip. It is advisory only (never a correctness input — the kernel, first-merge-wins, and the #3164 lock remain the backstops), preserves the ADR-010 affinity/gap order within each shard, and leaves the canonical write-access selection path byte-for-byte unchanged. Sharding *reduces* (not eliminates) cross-fork collisions — the residue is what the still-evidence-gated Phase-2 lease (SPEC-053-A §8.4) would remove if the ADR-070 metric ever justifies it.
