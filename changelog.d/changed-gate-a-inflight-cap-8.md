Lowered the default coordinated-`--prove` in-flight cap `UNSORRY_MAX_GATE_A_IN_FLIGHT` from 20 to **8** to match the actual Namespace verifier capacity (ADR-058). The pool is ~10 runners split across the prepare/audit/replay lanes (4/4/2), but the governor admitted up to 20 concurrent Gate A runs — roughly 2× the machines. The over-subscription made Gate A jobs get **cancelled** under contention rather than finishing, which stranded otherwise-valid proof PRs as un-mergeable "failed Gate A" (then re-queued, compounding the load). Capping in-flight near capacity lets checks **complete** instead of cancelling, so *useful* merge throughput goes up even though peak concurrency goes down. Still env-overridable (raise it when more runners are added; `-1` disables the limit).
