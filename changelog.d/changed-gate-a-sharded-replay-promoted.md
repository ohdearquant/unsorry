The required `gate-a` kernel replay now **shards across N parallel runners** ([ADR-063](docs/adrs/ADR-063-Sharded-Gate-A-Kernel-Replay.md) / [SPEC-063-A](docs/adrs/specs/SPEC-063-A-Sharded-Gate-A-Kernel-Replay.md) §5, #1909), after the `gate-a-shard-pilot` validated it on real runners. The single serial `gate-a-replay` job became three: `gate-a-replay-plan` (partitions the replay target set into N disjoint shards), a `gate-a-replay` **matrix** (one leg per shard, each restoring gate-a-prepare's oleans and kernel-replaying its slice — skipped entirely when no library module changed), and `gate-a-replay-cover` (fails closed unless the plan succeeded and every shard is green, i.e. every olean replayed exactly once). This cuts the ~1-hour full-replay long pole to ~1/N wall-clock so Gate A keeps pace with the submission rate. Soundness is unchanged — the partition is unit-tested disjoint+covering, shards share only the git SHA (leanchecker inputs stay locally-derived, ADR-049), and the daily `gate-a-full-replay` backstop is retained. The shard count is the operator knob `vars.UNSORRY_REPLAY_SHARDS` (default 8); the required context name `gate-a` is unchanged (ADR-058).
