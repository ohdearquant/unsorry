The post-merge **leaderboard** refresh now recomputes the corpus **once** per run instead of twice (ADR-082 / SPEC-082-A). The workflow previously ran a standalone `tools.leaderboard --check` drift probe and then a separate `--write` — two ~10-min full-corpus recomputes per refresh over the growing active+archive proof set (ADR-041). Under the swarm's merge firehose that doubled the window in which `main` advances mid-refresh, so the published `docs/metrics/leaderboard-ui.json` (and the live unsorry-guild proofs graph that reads it) lagged the latest merges by ~25–30 min, and it wasted CI runner minutes. A new single-pass `tools.leaderboard --write-if-stale` mode computes the seven generated artifacts exactly once, writes them only on drift, and signals drift via its exit code (`1` = was stale and rewritten, `0` = already in sync) — sharing one staleness/write definition with `--check`/`--write` (DRY). The cheap-push-retry/rebase loop (#426), the trigger model, and the artifacts produced are unchanged; the change is soundness-neutral (generated human-facing artifacts only). Halves the hot-path compute and the mid-refresh staleness window. New CLI mode is unit-tested for drift/no-op/idempotency/mutual-exclusivity.
