Fixed `gate-a-replay` shard OOM on 16 GB runners: the sharded kernel-replay job (ADR-063) used the default `UNSORRY_REPLAY_CHUNK` (30), so `leanchecker` — which holds ~all of mathlib (~10 GB) resident per process — peaked past 16 GB when a shard replayed a large changed set (e.g. a forced-full or bulk-archive run). Namespace runners block `swapon`, so the best-effort swap step provides no fallback there. The job now caps the per-invocation peak (`UNSORRY_REPLAY_CHUNK`, default 6, tunable via the repo var), matching `gate-a-full-replay`; each `leanchecker` invocation holds mathlib + a few oleans and fits in RAM.
