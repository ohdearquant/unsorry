Reconciled `swarm/run.sh`'s "run exactly ONE dispatcher" guidance with the scheduled `queue-dispatcher` workflow (which postdates run.sh): when that workflow is enabled it already _is_ the one dispatcher, so launching `run.sh` adds a second. ADR-064 goal-level dedup keeps this mostly harmless — both dispatchers read the same open-PR set and skip a goal already proved or already PR'd — but two passes can still race to open a duplicate PR within the window before one is visible to the other (first-merge-wins then closes the loser as a conflict, wasting a Gate A slot). `swarm/run.sh`, `CONTRIBUTING.md`, `swarm/README.md`, and `docs/recovery.md` now all say: on a repo with the scheduled dispatcher active, run a prover only (`./swarm/supervise.sh --prove`) rather than `run.sh`; use `run.sh` for a standalone/forked deployment with no scheduled dispatcher. Surfaced by an end-to-end run.sh test that observed its local dispatcher and the scheduled one opening PRs concurrently in the same window.
