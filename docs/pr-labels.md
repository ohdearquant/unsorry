# PR labelling and title conventions

Every PR carries labels classified from its title — the titles are machine-generated by the swarm or conventional-commit shaped, so labelling is deterministic. The classifier is `tools/repo/pr_labels.py` (single source of truth); `.github/workflows/pr-labels.yml` applies it on PR open/edit, and the same classifier drove the retroactive pass over the historical PRs.

A scheduled backstop, `.github/workflows/pr-label-sweep.yml`, runs the same classifier every 15 minutes over any open PR that still has **no** label and applies it. This catches the case where a per-PR `pr-labels` run is queued-out, superseded, or cancelled under a burst of PRs (the swarm or contributors opening dozens at once) and the PR would otherwise stay unlabelled forever. Both paths share one classifier, so the rules live in exactly one place; a title matching no known shape is left for the `pr-conventions` gate to flag, not silently labelled.

**The title shape is enforced (ADR-026).** The `pr-conventions` check (`.github/workflows/pr-conventions.yml`) runs `python3 -m tools.repo.pr_labels enforce "<title>"` and **fails any PR whose title matches no known shape**. Title discipline is the contract: fix the title, not the classifier.

| Label | Applied to (title shape) | Meaning |
|---|---|---|
| `swarm:translate` | `tr(<goal>): …`, `converge(<goal>): …` | Agent translation/convergence PR (Phase-0/1 loop) |
| `swarm:prove` | `prove(<goal>): …` | Agent proof PR — **theorem proved** (the product surface) |
| `swarm:decompose` | `decompose(<goal>): …` | **Theorem not proved** — split into sub-lemmas (ADR-009) |
| `swarm:demote` | `affinity(<goal>): …` | **Theorem not proved** — attempt failed, goal demoted (ADR-010) |
| `red-team` | `redteam<round>(<vector>): …` | Adversarial gate-bypass attempt — a **closed** red-team PR means the gates held |
| `release` | `docs(vX.Y.Z): …` | Version housekeeping + tag |
| `docs` | `docs: …` | Documentation, ADRs, specs, proposals |
| `metrics` | `docs: … (run NNN)` / `… metrics …` / `… red-team round NNN` | Run-evidence records (always alongside `docs`) |
| `feat` / `fix` / `chore` / `ci` / `test` / `refactor` / `perf` / `build` | Conventional-Commits prefixes (scope optional, `:` required) | Maintainer changes to the machinery |

So a glance at the title says what a PR *is*: `prove(...)` is a theorem that **passed**; `decompose(...)` / `affinity(...)` are theorems that **did not pass** (split / demoted); `feat:`/`fix:` are the machinery evolving; `docs:` is documentation.

**One logical change per PR (trunk-based).** A proof is a proof; a fix is a fix; a feature is a feature — don't bundle (e.g. a harness fix riding a proof PR). One short-lived branch off `main`, squash-merged on green gates, deleted after. See [`CONTRIBUTING.md`](../CONTRIBUTING.md) and [ADR-026](adrs/ADR-026-PR-Convention-Enforcement.md). Further separation (blocking PRs that mix proof and harness content) and harness-regression gates are tracked in issue #302.

Reading the history through labels: `swarm:*` PRs are the swarm working; `red-team` PRs closed-unmerged are the gates being attacked and holding; `metrics` PRs are the evidence trail; `feat`/`fix` are the machinery evolving.
